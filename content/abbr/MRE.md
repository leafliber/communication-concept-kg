### 英文全称

- Measurement Report Event-based

### 中文释义

- 基于事件上报的测量报告

### 解释

#### **为什么需要 MRE**（基于事件的 MR）

如果手机只进行**周期性上报**（Periodic MR），可能会错过关键的切换时机或产生大量无用数据：

- **场景 A**（漏报风险）：手机快速驶入弱覆盖区，若下一个周期上报时间点还没到，呼叫可能已经掉话了。
- **场景 B**（资源浪费）：手机静止在信号极好的地方，每秒上报一次完全相同的“信号好”数据，浪费空口资源和基站处理算力。

**MRE 的机制**解决了这个问题："**没事别打扰我，出事了**（或有机会了）"

#### **2. 核心触发事件** (3GPP 定义的标准事件)

在 LTE 和 5G NR 中，3GPP 定义了标准的 **A1 ~ A6** (LTE) 和 **B1 ~ B2** (异系统) 等事件。当测量结果满足这些事件的“进入条件”并持续一段时间（Time to Trigger, TTT）后，UE 就会上报 **MRE**。

表格

|事件 ID|名称|触发条件含义|典型应用场景|
|---|---|---|---|
|**A1**|服务小区好于门限|Serving > Threshold|**停止测量/省电**：信号太好了，告诉基站“我可以减少邻区测量了”，节省手机电量。|
|**A2**|服务小区差于门限|Serving < Threshold|**启动测量/准备切换**：信号变差了，告诉基站“快帮我看看有没有更好的邻居”，触发异频/异系统测量。|
|**A3**|邻区比服务小区好 (Offset)|Neighbor > Serving + Offset|**同频切换**：最经典的切换事件。发现旁边有个基站信号比我现在的强 enough，请求切换。|
|**A4**|邻区好于门限|Neighbor > Threshold|**负载均衡/异频切换**：发现某个特定频段的邻区信号很好，即使当前服务小区还行，也可以切过去分担负载。|
|**A5**|服务小区差 + 邻区好|Serving < Thresh1 AND Neighbor > Thresh2|**紧急切换/覆盖补救**：当前信号很差，且发现了一个足够好的邻区，用于快速挽救即将掉话的连接。|
|**A6**|邻区比辅小区好|Neighbor > SCell + Offset|**载波聚合** (CA)：在双连接场景下，优化辅小区的配置。|

_(注：B1/B2 事件通常用于切换到非 3GPP 系统，如 WLAN 或 2G/3G)_

#### **3. MRE 与 周期性 MR 的对比**

表格

|特性|**周期性 MR**(Periodic)|**基于事件的 MR**(MRE)|
|---|---|---|
|**触发机制**|定时器驱动 (如每 480ms, 1s, 2s)|**事件驱动** (满足 A1-A6 条件)|
|**数据特点**|连续、完整，能描绘信号变化轨迹|**离散、突变**，只记录关键转折点|
|**主要用途**|**覆盖分析、干扰定位、MR 定位** (需要连续采样)|**移动性管理** (切换)、参数优化、掉话分析|
|**空口开销**|较高 (无论信号好坏都发)|**较低** (仅在状态变化时发)|
|**对网优的意义**|用于画热力图、找盲点|用于分析**切换成功率**、**乒乓切换**、**晚切换/早切换**|


#### **4. MRE 在网优中的实际应用**

虽然标准里不叫 "MRE 文件"，但在分析切换问题时，工程师会专门筛选出**带有 Event ID 标记的 MR 记录**进行分析：

1. **切换失败定界**：
    
    - 查看是否上报了 **A3/A5 事件** (MRE)。
    - 如果上报了 MRE 但切换失败：可能是目标基站拒绝、传输问题或信令交互超时。
    - 如果**没上报 MRE**：说明参数设置不合理（如迟滞 Hysteresis 太大、TTT 太长），导致手机觉得“还不够好”，迟迟不触发切换，最终导致弱覆盖掉话（**晚切换**）。
2. **乒乓切换优化**：
    
    - 如果在短时间内，同一对小区间频繁交替触发 **A3 事件** (MRE)，说明两个小区覆盖重叠过大或切换门限设置过灵敏。
    - 优化手段：调整 A3 事件的 Offset 值或 TTT 时间。
3. **异频/异系统策略验证**：
    
    - 检查 **A2 事件** (启动异频测量) 的门限是否合理。设得太高，手机过早开启耗电测量；设得太低，手机来不及测就掉话了。

#### **5. 总结与关联**

- **概念澄清**：**MRE** 是工程上对 **"Event-triggered Measurement Report"** 的简称，强调其**触发机制**。
- **与 MR 的关系**：MRE 是 MR 的一个**子集**。所有的 MRE 都是 MR，但不是所有的 MR 都是 MRE（还有周期性的）。
- **与 MRO 的关系**：MRO 是原始数据集合，其中既包含周期性 MR，也包含基于事件的 MR (MRE)。在分析切换问题时，我们主要从 MRO 中提取 MRE 类记录。
- **与 3GPP 的关系**：3GPP 定义了具体的 **Event A1-A6** 逻辑，设备商据此实现 MRE 功能。
