### 英文全称

- Hybrid Automatic Repeat reQuest

### 中文释义

- 混合自动重传请求

### 解释

HARQ 是无线通信中结合了前向纠错编码 (FEC) 和自动重传请求 (ARQ) 的纠错机制，是 LTE 和 5G NR 实现高可靠性数据传输的核心技术。

### HARQ 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                     HARQ 工作流程                               │
│                                                                 │
│  发送端                    接收端                                │
│    │                         │                                   │
│ 1. │─────── 数据 + FEC ─────→│  发送数据（含冗余信息）              │
│    │                         │                                   │
│ 2. │←─────── ACK/NACK ───────│  解码成功 → ACK                   │
│    │                         │  解码失败 → NACK                   │
│    │                         │                                   │
│ 3. │────── 增量冗余 (重传) ──→│  使用前次+本次冗余联合解码           │
│    │                         │                                   │
│    ... (循环直到 ACK 或 最大重传次数)                            │
└─────────────────────────────────────────────────────────────────┘
```

### FEC + ARQ = HARQ

| 技术 | 作用 |
|------|------|
| **FEC (前向纠错)** | 接收端直接纠错，无需重传（如 Turbo Code、LDPC） |
| **ARQ (自动重传)** | 接收端检测错误，请求重传（如 CRC 校验） |
| **HARQ (混合)** | 两者结合，兼顾可靠性与效率 |

### HARQ 类型

| 类型 | 说明 | 应用 |
|------|------|------|
| **Type I (Chase Combining)** | 重传相同的原始数据包，多次合并提高信噪比 | LTE |
| **Type II (IR, Incremental Redundancy)** | 重传不同的冗余信息，软合并逐步提高可靠性 | LTE/5G NR |
| **Type III** | 重传时可单独解码的 IR | 5G NR |

### 5G NR HARQ 特点

| 特性 | LTE | 5G NR |
|------|-----|-------|
| **HARQ 进程数** | 最大 8 个 | 最大 16 个 (FDD), 更多 (TDD) |
| **反馈时机** | 固定 4ms 后 | 灵活配置 (动态调度) |
| **异步 HARQ** | 不支持（同步） | 支持（异步重传） |
| **上行 HARQ** | 必须 | 可配置为异步 |
| **URLLC 增强** | 标准 HARQ | Mini-slot 级 HARQ (<1ms) |

### HARQ 在 5G URLLC 中的作用

[[URLLC]] 场景对 HARQ 有特殊要求：

| 需求 | 实现方式 |
|------|---------|
| **超低时延** | Mini-slot 调度（2/4/7 个 OFDM 符号） |
| **高可靠性** | 额外的 PDCCH 确认、重复传输 |
| **配置** | Grant-Free 调度，无需每次请求 |

### 上行 HARQ 流程

| 步骤 | 说明 |
|------|------|
| 1 | UE 通过 PUSCH 发送数据 |
| 2 | gNB 检测并解码 |
| 3 | 解码成功 → UE 收到 [[PDSCH]] 上的 ACK |
| 4 | 解码失败 → UE 收到 NACK，触发重传 |
| 5 | UE 在调度资源上重传（增量冗余） |

### HARQ 与 [[QoS]] 的关系

| QCI/5QI | HARQ 优先级 | 配置 |
|---------|------------|------|
| URLLC (65/68) | 最高 | Mini-slot HARQ，低时延调度 |
| VoNR (1) | 高 | 标准 HARQ |
| eMBB (8/9) | 中 | 标准 HARQ，可配置冗余版本 |

### 相关概念

- [[NR]] - HARQ 是 NR 物理层的核心机制
- [[RLC]] - RLC 提供与 HARQ 的协同
- [[PDSCH]] / [[PUSCH]] - HARQ 传输的物理信道
- [[URLLC]] - URLLC 对 HARQ 有特殊要求
- [[QoS]] - 不同业务配置不同的 HARQ 策略

> 反向链接: [[RLC]] 的 ARQ 机制与物理层的 HARQ 形成双重保护，[[MAC]] 层负责调度 HARQ 重传资源，两者共同保障数据传输的可靠性
