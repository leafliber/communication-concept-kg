### 英文全称

- Minimization of Drive Tests

### 中文释义

- 最小化路测

### 解释

#### **核心定义与背景**

- **传统痛点**：在 MDT 出现之前，运营商为了评估网络覆盖和质量，必须派遣工程师驾驶测试车，携带专业的测试终端和扫描仪，沿着道路进行人工路测（Drive Test, DT）。
    - **缺点**：成本高昂、效率低、只能覆盖道路（无法测试室内）、数据非实时、存在安全隐患、无法获取海量用户样本。
- **MDT 的解决方案**：MDT 技术允许运营商**配置商用手机**（UE）在后台自动测量无线信号，并将带有**地理位置信息**的测量报告上报给网络。
    - **核心价值**：利用海量真实用户的手机代替专业的测试车，实现**低成本、全覆盖**（含室内）的网络质量评估。

#### **2. MDT 的两种主要模式**

根据数据采集的触发方式和实时性，[[3GPP]] 标准定义了两种模式：

| 模式       | **Logged MDT**(记录型)                                 | **Immediate MDT**(即时型)        |
| -------- | --------------------------------------------------- | ----------------------------- |
| **适用状态** | **RRC_IDLE** (空闲态) 或 **RRC_INACTIVE**               | **RRC_CONNECTED** (连接态)       |
| **工作机制** | 终端在空闲时按配置记录数据（含位置），存储在本地；当终端再次进入连接态时，将历史记录**批量上报**。 | 终端在连接状态下，**实时**测量并立即上报给基站。    |
| **主要用途** | **覆盖优化**：发现弱覆盖区域、覆盖空洞（尤其是室内和行人区域）。                  | **移动性优化**：切换参数调整、干扰分析、实时故障定位。 |
| **位置精度** | 通常依赖 GNSS (GPS)，精度较高；若无 GPS 则基于 RF 指纹。              | 可结合 GNSS 或网络侧定位技术。            |
| **数据延迟** | 高（取决于用户何时重连）。                                       | 低（准实时）。                       |

#### **3. MDT 数据包含的关键信息**

一条有效的 MDT 记录通常包含：

1. **无线测量量**：服务小区和邻区的 RSRP (信号强度), RSRQ (信号质量), SINR (信噪比)。
2. **位置信息** (Location Info)：
    - **GNSS 坐标**：如果终端支持且开启 GPS，提供高精度经纬度。
    - **RF 指纹**：如果没有 GPS，通过测量多个小区的信号特征来估算位置。
3. **时间戳**：精确的测量时间。
4. **上下文信息**：小区 ID (ECGI/NCI), TA (时间提前量), 终端类型，甚至包括业务体验指标（如 QoE，视频卡顿率等，需增强型 MDT 支持）。

#### **4. MDT 与 MR** (Measurement Report)

这是最容易混淆的两个概念，它们既有联系又有本质区别：

|维度|**MR**(测量报告)|**MDT**(路测最小化数据)|
|---|---|---|
|**主要目的**|**网络控制**：用于切换决策、链路自适应、功率控制等实时业务流程。|**网络优化**：用于事后分析、覆盖规划、参数验证，**不参与**实时控制。|
|**位置信息**|**通常无**：标准 MR 一般只带 Cell ID，不带经纬度（除非是定位专用 MR）。|**必须有**：MDT 的核心特征就是**携带地理位置信息** (GNSS 或估算坐标)。|
|**采集状态**|仅在 **连接态** (Connected) 下发生。|可在 **空闲态** (Idle) 和 **连接态** 下发生 (Logged MDT 覆盖空闲态)。|
|**采样密度**|高频、连续（为了快速响应信道变化）。|低频、稀疏（为了节省终端电量和空口资源，按策略采样）。|
|**数据流向**|UE -> 基站 -> (实时处理/统计)。|UE -> 基站 -> **TCE** (Trace Collection Entity) / 大数据平台 -> 网优系统。|

**一句话区分**：**MR** 是基站用来“开车”的（实时控制），**MDT** 是网优工程师用来“画地图”的（离线优化）。

#### **5. MDT 的处理架构** (基于 [[TMN]]/3GPP)

1. **配置下发**：
    - **EM/OSS** (网管系统) 生成 MDT 配置任务（如：在某个区域、针对某类终端、采集多长时间）。
    - 配置通过 **MME/AMF** (核心网控制面) 下发给 **eNodeB/gNB** (基站)。
2. **数据采集**：
    - 基站将配置通过 RRC 信令发给符合条件的 **[[UE]]**。
    - UE 在后台静默执行测量和记录。
3. **数据上报**：
    - UE 将数据上报给基站。
    - 基站通过 **Trace** 接口将数据转发给 **TCE** (Trace Collection Entity，跟踪收集实体)。
4. **分析与可视化**：
    - [[TCE]] 将数据解析并传递给 **NMS/OSS** 中的网优模块。
    - 系统将 MDT 点投射到地图上，生成**自动路测图**，替代传统的路测软件生成的轨迹图。

#### **6. MDT 的应用价值**

- **室内覆盖评估**：传统路测车进不去大楼，MDT 可以利用楼内用户的手机数据，绘制室内覆盖热力图。
- **深度覆盖分析**：发现地下室、电梯、偏远农村等传统手段难以触及的盲点。
- **参数验证闭环**：调整了天线倾角或切换参数后，无需再次出动车队，直接观察次日 MDT 数据的变化即可验证效果。
- **QoE** (体验质量)：增强型 MDT (eMDT) 可以关联应用层的体验指标（如视频缓冲时间、网页加载时长），实现从“信号好”到“体验好”的跨越。

#### **7. 挑战与隐私**

- **终端耗电**：频繁开启 GPS 和测量会增加手机耗电，因此 MDT 配置通常非常保守（低频率、特定场景触发）。
- **用户隐私**：MDT 涉及用户轨迹。
    - **合规要求**：必须经过用户同意（通常在入网协议中默认勾选，但需符合 GDPR 等法规），且数据在传输和存储过程中必须进行**匿名化处理**（去除 IMSI 等直接标识符）。
- **样本偏差**：MDT 数据依赖于“有用户在且开启了功能”的场景。如果某个区域没人用网，就没有 MDT 数据（这与路测车可以主动去测不同）。
